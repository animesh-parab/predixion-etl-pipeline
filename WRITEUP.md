# Business & Architecture Write-Up

## Executive Summary

This ETL pipeline bridges the gap between raw, noisy logs generated by Predixion AI's voice agents (VOIZ, KOLLECT) and the structured reporting layer required by business stakeholders. The primary objective was to ensure data integrity so that downstream BI tools and operational dashboards report accurate connect rates, SLA metrics, and financial outcomes without breaking due to malformed JSON payloads.

---

## Key Design Choices & Business Rationale

**1. Idempotent Database Loads (`INSERT OR REPLACE`)**

- **The Choice:** The pipeline uses `call_id` as a primary key with an upsert/replace mechanism to load data into the local SQLite database.
- **Business Rationale:** Call center operations frequently require backfilling data or re-running pipelines due to upstream network drops. Idempotency guarantees that executing this script multiple times will never duplicate records. This prevents artificially inflating an agent's call volume or double-counting `amount_promised`, which would corrupt performance evaluations and revenue forecasting.

**2. Chronological Validation & Cascading Rejections**

- **The Choice:** Records are validated sequentially — checking for structural integrity (missing fields), then domain validity (malformed timestamps), and finally duplication.
- **Business Rationale:** By categorizing rejections logically rather than just dropping them silently, the resulting `rejected_log.csv` acts as a diagnostic tool for the Data Engineering team to identify upstream bugs in the VOIZ/KOLLECT event emitters.

**3. Intentional Imputation (`is_amount_imputed` flag)**

- **The Choice:** Null values in `amount_promised` are imputed to `0`, but crucially, flagged with a boolean `is_amount_imputed = True`.
- **Business Rationale:** Null financial values break standard SQL `SUM()` and `AVG()` aggregations in BI tools like Tableau or Metabase. Imputing to zero ensures dashboards render cleanly, while the boolean flag allows analysts to distinguish between "$0 promised" and "no data collected," preserving the integrity of intent-to-pay models.

**4. Timezone Normalization (IST)**

- **The Choice:** All call timestamps are strictly normalized to Indian Standard Time (IST) before duration calculations.
- **Business Rationale:** Workforce management and SLA tracking rely entirely on localized shift schedules. If UTC timestamps bleed into the reporting layer, a callback requested at 4:00 AM IST might be erroneously attributed to the previous day's shift, disrupting staffing models.

---

## What I Would Do Differently at Scale

While Python standard libraries, Pandas, and SQLite are sufficient for this 500-record local simulation, scaling this to process millions of concurrent VOIZ/KOLLECT agent events requires a robust cloud-native architecture:

1. **Storage (Data Lakehouse):** I would deprecate SQLite entirely. Raw logs would land in AWS S3 or GCS. The transformation layer would write clean records as columnar Parquet files, partitioned by `call_date` to optimize analytical query costs and speed.

2. **Transformations (dbt over Pandas):** Relying on in-memory Pandas dataframes creates memory bottlenecks at scale. I would shift the transformation logic down to the data warehouse (e.g., Snowflake or BigQuery) and manage the SQL models using **dbt (Data Build Tool)**. This provides built-in testing, lineage graphs, and version control for the business logic.

3. **Orchestration:** Instead of relying on a manual CLI execution, the pipeline would be scheduled and monitored via **Apache Airflow**. This allows for automated dependency management, automatic retries on failure, and SLA alerting.

4. **Streaming Ingestion vs. Batch:** Voice agent events are time-sensitive. If the business requires real-time dashboarding for live agent interventions, the architecture should pivot from daily batch ETL to a streaming architecture using Apache Kafka to consume events and Apache Flink to process durations and metrics on the fly.
